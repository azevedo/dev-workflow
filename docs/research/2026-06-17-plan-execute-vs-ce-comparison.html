⚖︎ Verdict

Both are mature, decisions-first pipelines with near-identical philosophies (decisions-not-code, research-before-structure, scope discipline). CE is broader and more harness-portable with deeper review-gating machinery (residual gate, tiered escalation, confidence-deepening) and dual-format output; dev-workflow is tighter on execute + review-apply (LoC tripwire, slice contract, code-shape label, reconcile + verify-then-keep guard). Both capture learnings (ours /ba:compound, CE ce-compound) and consume docs/solutions/.

▲ Where we're ahead within plan→execute; not audited vs all ~40 CE skills

  • Pre-slice LoC scope tripwire (T=2×Est, pause on M≥T) — CE has no quantitative scope-creep guard before code is written.
  • Slice / MR decomposition — /ba:slice yields ≤150-LoC self-contained slices; CE has units but no MR-sizing lens.
  • Code-shape decision label — explicit **Code-shape decision:** discriminator that slice + executor both consume; CE relies on prose framing.
  • Review apply-guard — bidirectional reconciliation (under/over-application) + verify-then-keep with auto-revert of newly-red fixes; CE's followup has no test-gated revert.
  • Mandatory convention-compliance gate — clears violations before a plan is written. But not really "ahead": we adapted this from CE's code-in-plans era; their current ce-plan dropped it (rationale not found). Reconsider — candidate to retire alongside slice + the LoC gate.
  • Explicit deviation protocol — Expected/Found/Why + round-numbered ## Deviations audit trail.
  • Backward-compat fence handling — pre-change plans treat all fences as literal so old plans don't under-size.
  • Cross-session handoff doc — /ba:handoff compacts git state + artifact paths + next steps. (CE has ce-sessions — verify overlap.)

◑ Design-stance differences philosophy, not features

  • Plan's purpose — CE: portable agent-agnostic decision artifact, progress derived from git · ours: executable checklist whose [x] state is the resume mechanism.
  • Code-in-plan control — CE: prose discipline ("directional guidance") · ours: machine-parseable **Code-shape decision:** label tools consume.
  • Scope control — CE: qualitative pre-research scope checkpoint · ours: quantitative pre-slice LoC tripwire + mandatory convention gate.
  • Delivery lens — CE: units land as atomic commits, sized by judgment · ours: explicit ship-time slice layer (≤150-LoC MRs) separate from "how to build".
  • Breadth vs depth — CE: domain-general, multi-harness (Claude/Codex/Gemini) · ours: code-only, Claude-Code-native, mechanically tighter within scope.
